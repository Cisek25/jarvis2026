## ⚠️ Krytyczne wnioski z sesji

1. **Zawsze sprawdzaj DOM przed pisaniem CSS** — założenia o strukturze (np. że "btn" wewnątrz card to ten sam co outer button) prowadzą do conflict reguł i ginących styli (lessons learned z v1.39-v1.40)

2. **Stare reguły CSS nadpisują nowe przy tej samej specificity** — przy każdym fix sprawdzaj `getMatchedCSSRules` lub manualny audit przez selectorText w document.styleSheets

3. **IdoBooking app.css to system "trap"** — wielu reguł z `!important` które nadpisują inline style JS. Trzeba odbijać specificity (chained classes, body.page-X).

4. **Klient liczy field jako całość** (label + input = 76px), nie sam input (52px). Przy alignment buttonów dopytuj — czy "wysoki jak input" czy "wysoki jak field".

5. **Klient testuje na różnych viewportach** — np. menu overflow pojawia się przy <1280px, Litepicker overflow przy <1280px. Zawsze test responsive.

6. **OG image z galerii klienta** (zamiast Unsplash) — eliminuje external dependency + brand-consistent. `/0/0/8.jpg` (Bulwary Odry) wybrane jako OG dla PL/EN/DE.

7. **Blog v4 to wzorzec do reuse** — IdoBooking systemowe Aktualności + nasz scraper /news. Klient pisze naturalnie w panelu, zero custom HTML. Zaimplementowane jako referencja na Apartamenty Parkowe (`client58154`).
